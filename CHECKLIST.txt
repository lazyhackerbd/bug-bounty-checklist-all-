# Bug Bounty Vulnerability Checklist

----------------------------------------------------------------

## 1. OAuth Vulnerabilities

### Redirect URI & Flow Manipulation
- [ ] Open redirect in OAuth chain (redirect_uri loosely validated)
- [ ] Path-based injection (/callback?next= abuse)
- [ ] Subdomain takeover chaining with OAuth redirect
- [ ] Redirect URI parameter pollution (multiple redirect params accepted)
- [ ] Double redirect leak (provider vs app-level redirect confusion)
- [ ] Case sensitivity bypass in redirect_uri
- [ ] Trailing slash normalization bypass
- [ ] URL encoding tricks on redirect_uri
- [ ] Wildcard *.domain.com abuse in redirect_uri

### State Parameter Attacks
- [ ] Missing or predictable state parameter
- [ ] State not bound to session
- [ ] Reused state tokens
- [ ] Missing server-side state validation
- [ ] State leakage via URL logs / referrers
- [ ] State exposed via analytics scripts

### PKCE Implementation Failures
- [ ] PKCE missing in public clients (mobile, SPA, native)
- [ ] code_verifier reused as state
- [ ] Verifier exposed in redirect URL
- [ ] Mismatch between challenge method (plain vs S256)
- [ ] PKCE not enforced server-side
- [ ] PKCE secret leaks via redirect

### Authorization Code Leakage
- [ ] Code in URL leaks (browser history, referrer headers, logs, analytics)
- [ ] Code interception via malicious extensions
- [ ] Code interception via XSS on callback page
- [ ] Code interception via proxy manipulation

### Flow Confusion Attacks
- [ ] Mix-up attacks (swapping identity providers)
- [ ] Confusing token audience
- [ ] Accepting token from wrong issuer
- [ ] Login vs connect confusion (same callback for login + account linking)

### Token Storage & Exposure
- [ ] Token in localStorage (XSS-prone)
- [ ] Token in URL fragments (#access_token)
- [ ] Insecure cookies (no HttpOnly/SameSite)
- [ ] Referer header token leaks
- [ ] Third-party script token leakage

### Scope & Consent Abuse
- [ ] Scope injection / escalation (modify scope parameter)
- [ ] Backend fails to validate requested scopes
- [ ] Over-permissioned OAuth apps (read profile also grants email, API, offline)

### Device Code / Social Engineering
- [ ] Device code phishing (attacker provides device code, victim enters on legit domain)

### Chainable Patterns
- [ ] OAuth + Open Redirect → code leak → token exchange → ATO
- [ ] OAuth + XSS on redirect page → steal code/state → impersonation
- [ ] OAuth + subdomain takeover → redirect_uri → full token capture
- [ ] OAuth + CORS misconfig → token exfiltration

## 2. 2FA / MFA Bypass

### Broken Gating / Logic Flaws
- [ ] 2FA not enforced after login (session marked authenticated without MFA)
- [ ] Direct access to /dashboard works without completing 2FA
- [ ] Only frontend checks `is2FA=true`
- [ ] API endpoints ignore 2FA state
- [ ] Backend trusts client flag (`"2fa_verified": false` → flip to true)
- [ ] No OTP validation server-side
- [ ] 2FA is cosmetic only

### OTP Verification Flaws
- [ ] No session binding (OTP sent to A, reused in B's session)
- [ ] Missing rate limiting on OTP
- [ ] OTP not tied to specific action (login, password reset, email change all use same)
- [ ] OTP reuse / replay (same code works multiple times)
- [ ] OTP brute force (4-6 digit codes)
- [ ] OTP sent to old email during email change
- [ ] OTP bypass via JSON array
- [ ] Null/empty OTP accepted
- [ ] Old OTP still valid

### Response/Status Code Manipulation
- [ ] Change 400 → 200 in response
- [ ] Modify "message":"Invalid OTP" → "OTP verified successfully"
- [ ] Change status codes (302 → 200)
- [ ] Modify JSON body (add "type":3 to bypass)
- [ ] Change boolean `false` → `true`
- [ ] Remove 2FA parameter entirely from request

### Password Reset Bypass
- [ ] Password reset flow skips 2FA
- [ ] Reset password → login without MFA prompt
- [ ] 2FA auto-disabled after password reset

### OAuth / SSO Bypass
- [ ] OAuth login (Google/GitHub) skips local 2FA entirely
- [ ] Inconsistent policy enforcement across auth paths
- [ ] OpenID misconfiguration to bypass 2FA

### Session-Based Bypass
- [ ] 2FA bypass using poor session management
- [ ] Pre-2FA session cookie still valid
- [ ] Cookie leaked before completing 2FA
- [ ] Activation/verification link used to login instead of OTP
- [ ] First login phase cookie replaces 2FA-issued cookie
- [ ] Session permission confusion (complete 2FA on your account, access victim's flow)

### "Remember Device" / Trusted Device
- [ ] `trusted_device=true` with no cryptographic binding
- [ ] Cookie reuse across sessions
- [ ] Token theft → permanent bypass

### Backup Code Weaknesses
- [ ] Predictable / sequential backup codes
- [ ] No one-time invalidation
- [ ] Stored in plaintext or weak hash
- [ ] Backup codes generated immediately after 2FA enabled, available on single request

### Race Conditions
- [ ] Parallel OTP validation requests accepted
- [ ] OTP validated multiple times before invalidation

### API / Endpoint Bypass
- [ ] Older API versions (/v1/, /v2/, /v3/) don't enforce 2FA
- [ ] Mobile app endpoints bypass 2FA
- [ ] Direct API calls after password step skip OTP

### Other 2FA Bypass Vectors
- [ ] 2FA disabled without re-authentication
- [ ] Enabling 2FA doesn't expire previous sessions
- [ ] Login via activation link bypasses 2FA
- [ ] Subdomain login bypasses 2FA (same creds, no OTP prompt)
- [ ] Referer header manipulation to bypass
- [ ] Remove "/code" from API endpoint in reset flow
- [ ] Change request method to bypass
- [ ] Brute-force OTP with ffuf
- [ ] Backup code generation for victim account
- [ ] Clickjacking/CSRF to disable 2FA
- [ ] 2FA page discloses sensitive info (phone number)

## 5. IDOR (Insecure Direct Object Reference)

### Basic Object Manipulation
- [ ] Change `user_id`, `account_id`, `order_id` in requests
- [ ] Change `/api/profile/123` to `/api/profile/124`
- [ ] Test all HTTP methods (GET, POST, PUT, PATCH, DELETE)

### Read-Based IDOR
- [ ] Access other users' personal info
- [ ] Access invoices, orders, chat messages
- [ ] Access private files, documents
- [ ] Enumerate sequential IDs for mass data extraction

### Write-Based IDOR
- [ ] Change email of another user
- [ ] Update password/settings of another user
- [ ] Modify billing details of another user

### Horizontal vs Vertical
- [ ] Horizontal: same role, different account
- [ ] Vertical: normal user → admin data/actions

### UUID / "Unguessable" ID Issues
- [ ] UUIDs leaked in responses
- [ ] Enumeration via API lists
- [ ] Predictable sequential fallback IDs exist alongside UUIDs

### File / Media IDOR
- [ ] Access private uploads via sequential IDs
- [ ] Download other users' files, images, invoices

### IDOR Bypass Techniques
- [ ] Add `%20`, `%09`, `%0b`, `%0c`, `%1c`, `%1d`, `%1e`, `/` to ID
- [ ] Try older API versions: `/api/v5/9` → `/api/v1/9`
- [ ] Wrap ID with array: `{"id":[1,2,3]}`
- [ ] Wrap ID with JSON object
- [ ] JSON parameter pollution
- [ ] Change ID case
- [ ] Path traversal in ID
- [ ] Change words (e.g., `/user/1` → `/users/1`)
- [ ] Add extension: `/users/01` → `/users/02.json`
- [ ] Try parameter pollution: `users=01&users=02`
- [ ] Use wildcards: `/users/01*` or `/users/*`
- [ ] Check Referer header used for validation
- [ ] Swap GUID with numeric ID or email
- [ ] Try GUIDs: `00000000-0000-0000-0000-000000000000`
- [ ] Use `X-Original-URL` / `X-Forwarded-For` headers
- [ ] Add `/details` or similar to the endpoint
- [ ] Try base64 encoded IDs

### GraphQL IDOR
- [ ] Nested object access abuse
- [ ] Change user ID in nested queries
- [ ] Deep object traversal for mass exposure

### JWT / Claims IDOR
- [ ] Backend accepts modified request param despite JWT having user_id

### Multi-Step / Business Logic IDOR
- [ ] View order → modify shipping → intercept delivery
- [ ] Change email → password reset → ATO
- [ ] Modify shared resource (group, team, org) → persistent impact

### Second-Order IDOR
- [ ] IDOR that triggers via export files, emails, or message alerts

### Blind IDOR
- [ ] Information disclosed via export files, emails, notifications

### IDOR in Password Reset / Email Flows
- [ ] Change another user's email
- [ ] Trigger password reset for victim

### Race Condition + IDOR
- [ ] Parallel updates on same object inconsistent

### Testing Workflow
- [ ] Use Comparer to compare identical requests from two accounts
- [ ] Use Match & Replace to automate ID swapping
- [ ] Try 403/401 bypass via Intruder (50-100 requests with different IDs)
- [ ] Check if 403/401 is thrown but action is still performed
- [ ] Look for encrypted IDs, try to decrypt
- [ ] GUID enumeration via Google Dorks, GitHub, Wayback, Burp history
- [ ] SignUp, Reset Password endpoints often disclose GUIDs
